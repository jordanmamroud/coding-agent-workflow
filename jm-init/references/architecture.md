# Architecture Reference

The single architecture this skill applies to every project. Read this fully before scaffolding or making placement decisions.

## Core principle

**Build code in the route that uses it. Promote to `src/` only when a second consumer is proven.**

When unsure where to put something, pick the option with the **smaller blast radius** — fewer routes affected if you're wrong.

This is a deliberate inversion of the "factor everything into shared layers upfront" approach. The reasoning: every guess about future structure has a cost when wrong, and you don't know what'll be reused until the second use case appears. Defer until evidence arrives.

---

## The two top-level folders

```
app/      Routes. Each folder is both a URL and a self-contained feature.
          Code lives in private subfolders (_components/, _lib/) that
          Next.js does not expose as routes.

src/      Stack-level infrastructure with no natural route owner. Only
          contains things that genuinely have nowhere else to live.
```

## Initial src/ structure (fixed for every project)

```
src/
  db/
    schema.ts        # Drizzle schema, single source of truth
    client.ts        # better-sqlite3 + drizzle client
    drizzle.config.ts
    migrations/      # generated by drizzle-kit
  lib/
    env.ts           # validated env vars (zod)
```

That is the entire `src/` tree at scaffold time. Everything else gets created on refactor day, only when proven necessary.

## Route folder structure

```
app/<route>/
  page.tsx                      # the route itself (Next.js convention)
  layout.tsx                    # only if this segment needs a custom layout
  loading.tsx                   # optional loading UI
  error.tsx                     # optional error boundary
  actions.ts                    # all server actions for this route in ONE file
  _components/                  # React components used only by this route
  _lib/                         # logic used only by this route
  [id]/                         # dynamic detail route, if applicable
    page.tsx
    actions.ts
    _components/
```

The underscore prefix tells Next.js these folders are not routes. They're private to the route.

## Folders that DO NOT exist at scaffold time

Do not create:
- `src/features/` — **ever**. The folder under `app/` is the feature.
- `src/domain/` — wait until refactor day proves it.
- `src/ui/` — wait until refactor day proves it.
- `src/hooks/`, `src/types/`, `src/utils/`, `src/services/`, `src/components/` — none of these.

Empty folders are guesses. Guesses confuse agents and humans both.

---

## Where new code goes (one rule)

For every new piece of code, ask in order:

1. UI used by one route → `app/<route>/_components/`
2. Logic used by one route → `app/<route>/_lib/`
3. Server action → `app/<route>/actions.ts` (always one file per route, multiple exports)
4. New table or column → `src/db/schema.ts`
5. New env var → `src/lib/env.ts`
6. "I might need this elsewhere later" → STOP. Put it in the route. Defer.

---

## Blast radius thinking

When unsure between two locations, pick the smaller blast radius.

Every file makes a claim about what other code will exist. Putting `format-cost.ts` in `src/lib/` claims "this will be widely used." If wrong, the file sits in the wrong place forever and confuses future placement decisions. Putting it in `app/runs/_lib/` claims only "the runs route uses this." If wrong about reuse, no harm done. If right, you move it later — `mv` plus import updates.

| Decision | Bigger radius (avoid when unsure) | Smaller radius (prefer) |
|---|---|---|
| New helper | `src/lib/foo.ts` | `app/<route>/_lib/foo.ts` |
| New component | `src/ui/Button.tsx` | `app/<route>/_components/button.tsx` |
| New type | `src/types/foo.ts` | inline in the file using it |
| New hook | `src/hooks/use-foo.ts` | `app/<route>/_lib/use-foo.ts` |
| Provider/adapter | `src/domain/providers/x.ts` upfront | `app/<route>/_lib/x.ts`, promote on second use |
| Splitting a file | Split into 3 files now | Wait until file >300 lines with clear sub-pieces |
| Adding an abstraction | Generic interface upfront | Concrete impl, abstract on second use |

## Promotion (when to move code from `app/<route>/` to `src/`)

Promote ONLY when one of these is true:

- **A second route already imports the same file** (proven shared)
- **`jscpd` flagged duplication across routes** (automated detection)
- **The file grew past ~300 lines AND has clear sub-pieces** (split first, possibly promote sub-pieces)

If none of these apply, leave it where it is.

### Promotion targets

- Generic utility (date, format, parse) → `src/lib/`
- Real business logic / orchestration → `src/domain/` (create folder on first promotion)
- Reused UI primitive (custom Button variant, shared composite) → `src/ui/` (create folder on first promotion)

shadcn/ui primitives belong to `components/ui/` (shadcn convention) — leave those alone, the skill installs them there at scaffold time.

### When NOT to promote despite duplication

`jscpd` flagged duplication is a candidate, not a mandate. Don't promote when:

- The duplicated code looks similar but represents different concepts (the classic DRY mistake — coincidental similarity, not shared meaning)
- One copy is in active flux; promoting would couple two things that should evolve separately
- The duplication is small (under ~10 lines) and extracting would create more import friction than it saves

---

## Stack-level infrastructure exception

Some things genuinely belong in `src/` from day one because they have **no natural route owner**:

- Database schema and client (`src/db/`)
- Validated environment config (`src/lib/env.ts`)
- Auth utilities, if auth exists
- Encryption helpers for secrets (when they're read/written across routes from day one)

The test: can you point to a single route that owns this code? If yes, defer. If no, it's infrastructure — `src/` is fine.

## Refactor day workflow

The deferred-promotion model only works if duplication is actually detected. Refactor days catch it.

### Setup (already done at scaffold time)

```bash
pnpm add -D jscpd knip
```

`package.json` scripts:

```json
"refactor-day": "pnpm jscpd && pnpm knip",
"jscpd": "jscpd --pattern \"app/**/*.{ts,tsx}\" \"src/**/*.{ts,tsx}\" --threshold 0 --min-lines 8",
"knip": "knip"
```

`.knip.json`:

```json
{
  "$schema": "https://unpkg.com/knip@latest/schema.json",
  "entry": ["app/**/page.tsx", "app/**/layout.tsx", "app/**/route.ts", "middleware.ts"],
  "project": ["app/**/*.{ts,tsx}", "src/**/*.{ts,tsx}"]
}
```

### The workflow

Run weekly (or after each major feature ships):

1. `pnpm refactor-day`
2. Review `jscpd` output — duplications between routes are promotion candidates
3. For each duplication, prompt an agent: *"jscpd found duplication between `app/X/_lib/Y.ts` and `app/Z/_lib/Y.ts`. Extract the shared part to `src/<appropriate-place>` and update both files. Atomic commit."*
4. Review `knip` output — unused exports/files are removal candidates
5. Commit each promotion as a separate atomic commit

### Cadence

- **Weekly** for active projects with multiple agents working in parallel
- **After each major feature** to catch the duplication that feature introduced
- **Before adding a third route** that you suspect will share logic with two existing ones

### Anti-patterns

- Don't run promotion in the middle of feature work. Different concerns, different time.
- Don't promote without seeing duplication. Trust the tooling.
- Don't add refactor day to CI. CI fails on bugs, not on shape. Refactor day is deliberate, human-gated cleanup.

---

## Why this works for parallel AI agents

Three properties matter when multiple agents work in parallel:

1. **One folder per task.** Agent gets a feature task, opens one folder, finds everything. No cross-folder traversal.
2. **Zero taxonomy decisions.** No "is this a widget or feature or entity?" — that judgment call is where agents get inconsistent. The architecture removes the question.
3. **Files don't overlap between agents.** Agent A on `app/runs/` and Agent B on `app/prompts/` literally cannot touch the same files unless they're both editing one of the documented collision-risk files (root layout, db schema, env).

The deferred-promotion model produces these properties naturally. Pre-abstracted architectures (FSD, layered, feature-first with `src/features/`) violate all three.

## Worked examples

**Q: New "Delete" button on the run results page.**
A: `app/runs/[id]/_components/delete-run-button.tsx`. Used by exactly one route. The action goes in `app/runs/[id]/actions.ts` as a `deleteRun` export.

**Q: Function that calls the OpenAI API.**
A: `app/runs/_lib/providers/openai.ts`. Smaller radius. If `app/settings/keys/_lib/validate-key.ts` later also calls OpenAI, jscpd flags it on refactor day → promote to `src/domain/providers/openai.ts`.

**Q: A `formatCurrency` helper. Surely shared?**
A: Maybe, but only after two routes use it. Until then, put it in the route that needs it.

**Q: Setting up a Postgres client.**
A: `src/db/client.ts`. There is one database. It has no route owner. Infrastructure exception applies.

**Q: `_components/` has 12 files. Split into subfolders?**
A: Probably not yet. 12 is fine. If it hits 25+ AND there are obvious clusters, then split into `_components/results/` and `_components/settings/`.

## The mental model

Three rules in priority order:

1. **Build in the route** that uses the code. Default for everything except infrastructure.
2. **Defer promotion** until you have proof: a second importer, jscpd flagging duplication, or a 300+ line file with clear sub-pieces.
3. **When unsure, smaller blast radius wins.** Smaller guesses are cheaper to undo.

Everything else — the AGENTS.md template, underscore folders, refactor-day tooling — is supporting machinery. The architecture itself is just those three rules.
